GOALS
- One-line paste install on Windows and Linux. Launch with `agent`.
- Self-contained runtime: nothing the program manages leaks onto the user's PATH or into global state.
- Fast launches (including agent-server)
- Update detection and apply, triggered from the Control Tower.

ARCHITECTURE RECAP
- TUI: Bun + React (OpenTUI). `bun build --compile` already produces standalone binaries (see dist/) that embed
  the Bun runtime, so end users need nothing preinstalled to run the TUI itself.
- agent-server: separate Python repo (github.com/DavidKoleczek/agent-server), run via uv. The TUI manages uv and
  uses it to install and run agent-server.

MANAGED RUNTIME DIRECTORY  [MRD]
- Single managed root: ~/.agents/tui/  (Windows: %USERPROFILE%\.agents\tui\). Everything the program owns lives
  here so uninstall is deleting one directory. No footprint on the user's PATH or global uv state.
- DONE: the root is exported as MANAGED_ROOT (src/lib/constants.ts), resolved from os.homedir() so one path serves
  both Windows and POSIX. The uv binary lives under it, and the cache/tools/bin/python dirs below are routed via the
  UV_* env vars on the uv spawn (see [UV]/[SRV]).
- Layout:
    uv/<uv-version>/uv[.exe]   managed uv binary, version-scoped
      cache/                     UV_CACHE_DIR     (wheel and build cache)
      tools/<sha>-py<ver>/       UV_TOOL_DIR      (per-pin agent-server tool environment)
      bin/<sha>-py<ver>/         UV_TOOL_BIN_DIR  (per-pin agent-server entry point; NOT added to the user's PATH)
      python/                    UV_PYTHON_INSTALL_DIR (managed interpreter)

UV MANAGEMENT  [UV]
- Always use our own pinned uv; never a system uv. Identical behavior in `bun dev` and the compiled exe.
- DONE: src/lib/server/lifecycle/uv.ts resolves uv from a version-scoped cache dir under the managed root
  (~/.agents/tui/uv/<uv-version>/), downloads only when missing or when the pinned version changes, and prunes old
  versions. The pinned version comes from PINNED_VERSIONS.uv (src/lib/versions.ts).
- DONE: uvSandboxEnv() (uv.ts) pins UV_CACHE_DIR, UV_TOOL_DIR, UV_TOOL_BIN_DIR, and UV_PYTHON_INSTALL_DIR to
  managed subdirs, spread into the uv spawn env after process.env (spawn.ts) so they override any user-set values.
  Verified via uv's introspection (uv cache dir / tool dir / tool dir --bin / python dir) that all four resolve
  under ~/.agents/tui and off uv's global defaults.

AGENT-SERVER RUNTIME  [SRV]
- Decision: use `uv tool install` (not ephemeral `uv tool run`) for faster warm launches. The persistent entry
  point is exec'd directly at launch, skipping per-run environment materialization. This targets the current
  server-startup perf issue.
- Pin agent-server to commit 46d15f83517c6b594149f7d2ff93e240695fa5c0 (PINNED_VERSIONS.agentServer) and Python to
  3.13 (PINNED_VERSIONS.python). uv treats a pinned commit as immutable, so after the first install, launches
  resolve from cache and work offline. Bump the SHA to take a newer agent-server.
- Each pinned (SHA, Python) installs into its own scope dir: tools/<sha>-py<ver> (UV_TOOL_DIR) and
  bin/<sha>-py<ver> (UV_TOOL_BIN_DIR). ensureAgentServer (agent-server.ts) gates on the scoped entry point
  existing: a warm launch exec's it without invoking uv; a new pin installs a new scope alongside the old one, so
  it never overwrites (and never needs --force against) a venv a running instance still has open. uv's own
  tool-dir lock serializes two launches racing the same pin.
- Sandbox via env vars on the uv spawn: UV_CACHE_DIR and UV_PYTHON_INSTALL_DIR are shared across pins; UV_TOOL_DIR
  and UV_TOOL_BIN_DIR are scoped per pin. All under ~/.agents/tui; verified these override uv's global defaults.
- Pin uv version AND Python version for reproducibility (UV_PYTHON_INSTALL_DIR keeps any managed interpreter
  inside the root rather than the user's global uv dir).
- Later (optional): vendor a per-platform wheelhouse published from agent-server CI for a fully offline first run
  (uv pip install --no-index --find-links). Larger cross-repo effort; not needed for the first cut.

VERSION PINS  [PINS]
- Single source of truth: a TypeScript module src/lib/versions.ts exporting a typed PINNED_VERSIONS object.
  Every consumer is Bun/TS (uv.ts, spawn.ts, the updater, the CI build script) and imports the value directly, so
  there is no runtime parsing and the values embed into the compiled binary.
- Holds the runtime pins we bump by hand: the uv version (DONE: PINNED_VERSIONS.uv, consumed by uv.ts), and later
  the agent-server commit SHA (currently GIT_REF="main" in spawn.ts) and the Python version. Add each only when a
  consumer wires it, so no pin sits in source unread.
- The app's own version is NOT in this module. It is injected at build time from the git tag via
  `bun build --define` (0.0.0-dev fallback in dev), since it changes every release.
- latest.json is DERIVED from this module, not a second copy: at release time CI reads PINNED_VERSIONS plus the
  injected app version and emits latest.json. One pin edit changes both runtime behavior and what an update
  advertises.

DISTRIBUTION AND INSTALL  [DIST]
- Host binaries on GitHub Releases. CI (GitHub Actions) cross-compiles the Windows and Linux x64 targets on tag
  push via `bun build --compile --target=bun-{linux,windows}-x64` (x64 only for now; arm64 later), bakes in the
  release version and pinned agent-server SHA via --define, and uploads assets plus a latest.json manifest.
- latest.json: app version, per-platform asset URLs, sha256 checksums, pinned agent-server ref, pinned uv version, pinned Python version.
  Generated in CI from the central pins module (see VERSION PINS) plus the build-injected app version. Fetched via
  the stable asset URL https://github.com/DavidKoleczek/agent-tui/releases/latest/download/latest.json, which
  always resolves to the newest non-prerelease and avoids GitHub API rate limits.
- Install scripts:
    Linux/macOS: curl -fsSL .../install.sh | bash
    Windows:     irm .../install.ps1 | iex
  Detect OS/arch, download, verify sha256, place as `agent` (Linux ~/.local/bin, Windows
  %LOCALAPPDATA%\Programs\agent-tui\), ensure that dir is on PATH.
- Optional: ship gzip/zstd-compressed assets (the binary compresses to roughly 40-50MB) to cut download time.

UPDATES  [UPD]
- In-place self-replace, the pattern used by bun/deno/rustup/uv. No separate launcher or trampoline process.
    Windows: move running agent.exe aside to agent.exe.old (allowed while running), write new agent.exe.
             The fresh process deletes agent.exe.old on its next start.
    Linux:   rename the new binary over the running one (the open inode keeps old code mapped until exit).
- After the swap the running TUI does NOT auto-relaunch: like bun/deno/rustup/uv it shows "Update installed,
  restart agent" and the user restarts. (Bun terminates spawned children when the parent exits on Windows, so
  an in-process relaunch cannot hand the console to the new binary; restart-to-apply is the reliable equivalent.)
- Trigger: Control Tower only for now. A manual check/update action fetches latest.json, compares against the
  baked-in version, and on confirm downloads + verifies sha256 + swaps.
- A TUI update also rolls the pinned agent-server SHA, uv version, and Python version, so the whole runtime
  updates in lockstep.
- The update step prunes superseded agent-server scope installs (old tools/<scope> and bin/<scope> dirs)
  best-effort, skipping any still in use by a running instance (a locked dir is left for a later run to reclaim).

UNINSTALL  [UNINST]
- `agent uninstall` tears down the managed runtime so a later install is clean (clean reinstall = uninstall, then
  re-run the install script). It is a CLI subcommand that does NOT boot the TUI or agent-server, so it holds no lock
  on the managed root itself.
- Precondition: no running instances. A live agent-server (its python worker, or the managed uv) keeps files under
  ~/.agents/tui open, and on Windows an open file blocks removing its directory. So uninstall first enumerates
  processes whose executable resolves under ~/.agents/tui (the exact set that would lock the delete: agent-server
  shims, python servers/workers, uv; via Win32_Process ExecutablePath on Windows, /proc/<pid>/exe on Linux). If any
  exist it aborts BEFORE touching anything and prints the offending PIDs/paths with a clear "quit those instances,
  then re-run `agent uninstall`" message. No partial teardown.
- Removal once the check passes:
    - The managed root ~/.agents/tui in full (uv, cache, python, tools/<scope>, bin/<scope>): the runtime, and the
      only state that affects a clean reinstall. Removed synchronously; the uninstall binary lives in
      %LOCALAPPDATA%\Programs\agent-tui (Windows) or ~/.local/bin (Linux), not under the managed root, so nothing it
      holds blocks this delete.
    - The installed `agent` binary, its install dir, and the PATH entry the installer added (HKCU\Environment\Path
      on Windows; the profile line on Linux, skipped when ~/.local/bin was already on PATH).
- Self-deletion of the running binary: a process cannot delete its own running image on Windows. Reuse the [UPD]
  self-replace primitive: spawn a short detached cleaner that waits for this PID to exit, then removes
  %LOCALAPPDATA%\Programs\agent-tui. On POSIX, unlink the binary directly (the open inode stays mapped until exit).
- Preserves user data: per-working-directory .agents/sessions (conversations) and .agents/logs live outside the
  managed root and are left untouched, so history survives an uninstall. The command output says so; `--yes` skips
  the confirmation prompt.

STATUS / SEQUENCING
Each step ends with the tag(s) of the section(s) above it implements: [MRD] [UV] [SRV] [PINS] [DIST] [UPD] [UNINST].
- DONE:  managed, version-scoped uv resolution (uv.ts).  -> [UV]
- DONE:  central pins module src/lib/versions.ts; uv version moved into PINNED_VERSIONS.  -> [PINS] [UV]
- DONE:  managed root at ~/.agents/tui (MANAGED_ROOT in constants.ts); uv binary migrated to it.  -> [MRD] [UV]
- DONE:  the four UV_* sandbox env vars set on the uv spawn, all under the managed root.  -> [MRD] [SRV]
- DONE:  pin agent-server SHA + Python 3.13; ensureAgentServer installs via `uv tool install` into per-pin scope
         dirs (warm launches skip uv) and spawn.ts exec's the scoped entry point.  -> [SRV] [PINS]
- DONE:  build script (scripts/build.ts) + latest.json generator (scripts/latest.ts); app version
         injected via bun build --define and exposed by `agent --version`; release CI
         (.github/workflows/release.yml) on release:published uploads assets via the REST API;
         install.sh / install.ps1 fetch latest.json, verify sha256, and place `agent` on PATH.  -> [DIST]
- DONE:  Control Tower update flow: src/lib/tui/update.ts (latest.json fetch + version compare, sha256- and
         size-verified download staged in the install dir, in-place self-replace swap, stale .old cleanup on
         next boot); "Check for updates" UpdatePanel on the Settings tab; on success shows "restart agent"
         rather than auto-relaunching (Windows kills spawned children on parent exit); ensureAgentServer
         prunes superseded agent-server scopes best-effort. Test end-to-end locally with `bun run e2e-update`.  -> [UPD]
- DONE:  `agent uninstall` clean-teardown command (src/lib/tui/uninstall.ts): a pre-boot CLI (boots nothing) that
         refuses while any process's executable resolves under the managed root, then removes the managed root, the
         installer's PATH entry, and the binary. Windows defers the install-dir delete to a detached cleaner spawned
         via WMI Win32_Process.Create (which outlives the process, unlike a Bun child); POSIX unlinks the running
         image directly. `--yes` skips the confirmation prompt. Per-directory .agents history is left untouched.
         Test end-to-end locally with `bun run e2e-uninstall`.  -> [UNINST]
